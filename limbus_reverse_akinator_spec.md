# Limbus 逆アキネーター 仕様書

## 0. 概要

- **タイトル**: Limbus 逆アキネーター
- **形態**: Web上で動く静的サイト（サーバーサイド処理なし。HTML/CSS/JSのみで完結）
- **ゲームの骨子**: システムが人格プール内から秘密の1体（正解人格）をランダムに選ぶ。プレイヤーは「質問一覧」から質問カテゴリを選び、値を入力して、その属性が正解人格と一致するかを尋ねる。返答は「はい/いいえ/不明」の3種類。プレイヤーは繰り返し質問して手がかりを集め、正解だと思ったタイミングで「回答」ボタンから人格を1つ選んで確定する。

---

## 1. 画面遷移

```
[タイトル画面] --開始--> [人格選択画面] --開始--> [ゲーム画面]
                                                    |
                                        新しいゲーム(同じ候補プールで再抽選)
                                                    |
                                              [ゲーム画面(リセット)]

ゲーム画面から「設定」を押すと人格選択画面に戻れる（候補プールの再編集）
```

### 1-1. タイトル画面
- 中央に「Limbus 逆アキネーター」の見出し
- 「開始」ボタン

### 1-2. 人格選択画面
- 全人格の一覧をチェックボックス付きリストで表示（デフォルト全チェック）
- チェックを外した人格はこのゲームの候補プール（＝正解になりうる人格、および質問一覧に影響する人格母集団）から除外される
- 「開始」ボタンでゲーム画面へ遷移。この時点でチェック済み人格からランダムに1体を正解として内部的に選出する

### 1-3. ゲーム画面（レイアウト）

```
┌─────────────────────────────────────────────┐
│ 設定                              新しいゲーム │
├───────────────────────┬───────────────────────┤
│                       │                       │
│      チャット欄        │      質問一覧          │
│  (Q&Aのやり取りが      │  (罪悪属性/攻撃タイプ/  │
│   時系列で積み上がる)   │   所属/速度/耐性/       │
│                       │   特性キーワード/       │
│                       │   スキル名 等)          │
│                       │                       │
├───────────────────────┼───────────────────────┤
│       入力欄           │        回答            │
└───────────────────────┴───────────────────────┘
```

- **設定**（左上）: 人格選択画面に戻る
- **新しいゲーム**（右上）: 同じ候補プールのまま正解人格を再抽選し、チャット履歴をクリア
- **チャット欄**（中央左・大きめ）: 質問と回答のやり取りをログ表示。時系列で下に積み上がる（自動スクロール）
- **質問一覧**（中央右）: 質問カテゴリのボタン/リスト。クリックすると入力欄がアクティブになり、そのカテゴリに対する値を入力できる状態になる
- **入力欄**（左下）: 質問一覧で選んだカテゴリに対する値をフリー入力するテキストボックス。Enterまたは送信で質問を確定しチャットに追加
- **回答**（右下）: 押すと人格を1体選ぶUI（検索付きドロップダウン等）が開く。選んで確定すると正誤判定が行われる
  - 正解 → 正解演出＋メッセージ（例: 使用した質問数を表示）。続けて「新しいゲーム」を促す
  - 不正解 → 「不正解、続行してください」的なメッセージをチャットに追加し、ゲーム継続（回答回数に制限は設けない）

## 2. 質問カテゴリと回答ロジック

### 2-1. 質問一覧に出すカテゴリ（例）

| カテゴリ | 入力形式 | 参照するデータ |
|---|---|---|
| 罪悪属性 | フリー入力（怠惰/傲慢/憤怒/暴食/色欲/憂鬱/嫉妬/怠慢のいずれか想定） | 全スキルのsin_typeの集合 |
| 攻撃タイプ | フリー入力（斬撃/貫通/打撃） | 全スキルのattack_typeの集合 |
| 所属 | フリー入力 | identities.affiliations |
| 特性キーワード | フリー入力 | identities.keywords |
| 速度 | 数値入力 | identities.speed_min ~ speed_max（範囲内かどうか判定） |
| 体力 | 数値入力 | identities.hp_min ~ hp_max（範囲内かどうか判定） |
| 耐性（斬撃/貫通/打撃） | フリー入力（耐性/普通/脆弱） | identities.resistances |
| パニックタイプ | フリー入力 | identities.panic_type |
| スキル名 | フリー入力（部分一致可） | skills.name |
| 実装シーズン | フリー入力 | identities.season |

*質問カテゴリの最終的な項目数・粒度は、用意するデータ内容に応じて実装側で微調整して良い。*

### 2-2. はい / いいえ / 不明 の判定ロジック

質問カテゴリごとに、入力値を**候補プール全体の既知の値（語彙）**と照合する。

- **既知の値と一致し、正解人格がその値を持つ** → 「はい」
- **既知の値と一致するが、正解人格はその値を持たない** → 「いいえ」
- **候補プール内のどの人格の値とも一致しない（＝未知の語・誤字・存在しない値）** → 「不明」

具体的な判定方式はカテゴリの型によって変わる：

| カテゴリの型 | 判定方法 |
|---|---|
| リスト一致型（所属/キーワード/罪悪属性/攻撃タイプ/耐性ラベル/パニックタイプ/シーズン） | 入力値を正規化（前後空白除去・全角半角統一など）し、候補プール全人格の当該フィールドの値集合と完全一致するか確認。一致する値が存在しなければ「不明」。一致する値が存在し、かつ正解人格がその値を持つなら「はい」、持たなければ「いいえ」 |
| 数値範囲型（速度/体力） | 入力が数値としてパース可能か確認。パース不可なら「不明」。パース可能なら、正解人格の範囲(min~max)に含まれるかで「はい/いいえ」を判定 |
| 部分一致型（スキル名） | 入力文字列が候補プール中のいずれかのスキル名に部分一致するか確認。一つも一致しなければ「不明」。一致する場合、正解人格のスキル名にも該当するか確認して「はい/いいえ」を判定 |

- チャット欄には「Q: 〈カテゴリ名〉は『入力値』ですか？」「A: はい / いいえ / 不明」のような形で表示する

## 3. データモデル（実装側に渡す想定のスキーマ）

静的サイトなのでバックエンドDBは使わず、**全人格データをJSONとしてバンドル**し、クライアントサイドJSで検索・判定する想定。

```json
{
  "id": "gregor_dawn_office",
  "character_name": "グレゴール",
  "title": "夜明事務所代表",
  "season": "ヴァルプルギス",
  "affiliations": ["夜明事務所"],
  "keywords": ["フィクサー", "夜明事務所"],
  "hp_min": 88, "hp_max": 271,
  "speed_min": 4, "speed_max": 6,
  "panic_type": "パニック",
  "resistances": [
    { "type": "斬撃", "label": "耐性" },
    { "type": "貫通", "label": "普通" },
    { "type": "打撃", "label": "脆弱" }
  ],
  "skills": [
    { "name": "黎明の閃光", "sin_type": "怠惰", "attack_type": "斬撃" },
    { "name": "茜さす剣", "sin_type": "傲慢", "attack_type": "斬撃" },
    { "name": "夜明けを裂く剣", "sin_type": "憤怒", "attack_type": "斬撃" }
  ]
}
```

※効果テキストの詳細（コイン効果・条件文など）はこのゲームでは不要。質問・判定に使う項目だけを持たせれば十分。

## 4. 非機能要件・技術メモ

- 完全に静的（バックエンド不要）。ホスティングもGitHub Pages等どこでも可
- 人格データJSONを外部ファイルとして読み込む形にしておくと、後からデータを追加・差し替えしやすい
- 状態管理はブラウザのメモリ内で完結（ページリロードでリセットされる想定。永続化が必要なら別途localStorage等を検討）
- モバイル対応は必須ではないが、チャット欄と質問一覧が横並びのレイアウトなので、狭い画面では縦積みにするなどのレスポンシブ対応があると望ましい

---

## 5. 未確定・実装側判断に委ねる事項

- 質問カテゴリの正確な項目リストと、入力補助（サジェスト表示など）の有無
- 「不明」判定の語彙マッチングの厳密さ（表記ゆれ許容範囲など）
- 回答UIの見た目（ドロップダウン/検索ボックス/一覧選択など）
- 正解時演出、質問数カウント表示の有無
- 設定画面で候補プール以外に調整できる項目（音量・テーマ等）の有無
